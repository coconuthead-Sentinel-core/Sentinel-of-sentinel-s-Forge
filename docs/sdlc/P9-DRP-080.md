## 1. Purpose and Objectives

This Disaster Recovery Plan defines the processes, roles, and technical procedures required to restore Sentinel Forge following a disruption. Objectives: (a) minimize business impact, (b) meet recovery targets (RTO/RPO), (c) ensure repeatable recovery through documented procedures, and (d) maintain data integrity.

---

## 2. Scope

**In scope:** Sentinel Forge application, Docker deployment stack, nginx proxy, Azure Cosmos DB, Azure OpenAI integration, Stripe webhook endpoints, CI/CD pipeline, monitoring endpoints.

**Out of scope:** Azure platform-level disasters (handled by Azure SLAs), Stripe platform availability, developer workstations, marketing/sales systems.

**Applicability:** Unplanned outages, container failures, database corruption, credential compromise, and deployment failures.

## 9. Recovery Procedures (Runbooks)

### 9.1 DR Invocation Criteria

| Trigger / Condition | Decision Guidance | Approver(s) |
|--------------------|--------------------|-------------|
| Application container down > 15 minutes | Restart container; if fails, redeploy from image | Shannon |
| Host machine failure | Provision new host; deploy from Git + container registry | Shannon |
| Database corruption | Initiate Cosmos DB point-in-time restore | Shannon |
| Credential compromise | Rotate all secrets immediately; redeploy | Shannon |
| Deployment failure (bad release) | Roll back to previous Docker image tag | Shannon |

### Scenario 1: Container Failure (RTO: 30 minutes)

1. **Detect:** Health check /healthz returns non-204 or container exits
2. **Triage:** Check Docker logs: `docker-compose logs app`
3. **Restart:** `docker-compose restart app`
4. **Validate:** Curl /healthz (expect 204) and /readyz (expect 200)
5. **If restart fails:** `docker-compose down && docker-compose up -d`
6. **Validate again:** Run scripts/smoke_test.py
7. **Escalate:** If still failing, check for config/env issues in docker-compose.yml

### Scenario 2: Database Corruption (RTO: 4 hours)

1. **Detect:** /readyz returns error; Cosmos DB queries failing
2. **Assess:** Determine scope of corruption via Azure portal
3. **Restore:** Initiate Azure Cosmos DB point-in-time restore to last known good state
4. **Update config:** Point application to restored database (if new endpoint)
5. **Validate:** Run database queries via /api/notes; verify data integrity
6. **Redeploy:** `docker-compose restart app`
7. **Smoke test:** Run scripts/smoke_test.py

### Scenario 3: Credential Compromise (RTO: 1 hour)

1. **Detect:** Suspicious activity in logs or external notification
2. **Contain:** Stop application: `docker-compose down`
3. **Rotate secrets:**
   - Generate new API_KEY
   - Generate new JWT_SECRET_KEY (invalidates all active tokens)
   - Rotate COSMOS_KEY in Azure portal
   - Rotate STRIPE_SECRET_KEY and STRIPE_WEBHOOK_SECRET in Stripe dashboard
   - Rotate AOAI_KEY in Azure portal
4. **Update config:** Update environment variables in deployment
5. **Redeploy:** `docker-compose up -d`
6. **Validate:** Run scripts/smoke_test.py
7. **Notify:** Inform affected users (if any) to re-authenticate

### Scenario 4: Rollback Bad Deployment (RTO: 30 minutes)

1. **Detect:** Tests failing or errors after deployment
2. **Decision:** If P0 impact, immediate rollback
3. **Rollback:** `docker-compose down`
4. **Revert:** `git checkout <previous-tag>`
5. **Rebuild:** `docker-compose build && docker-compose up -d`
6. **Validate:** Run scripts/smoke_test.py
7. **Investigate:** Debug the failed release in development environment

### Scenario 5: Failback to Primary (RTO: varies)

1. **Confirm:** Primary system is healthy and validated
2. **Sync:** Ensure data is current (Cosmos DB replication)
3. **Switch:** Update DNS/load balancer to point to primary
4. **Validate:** Run full smoke test
5. **Monitor:** Watch for 24 hours before declaring recovery complete
